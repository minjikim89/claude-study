# Agent Loop 해부 — 에이전트를 에이전트로 만드는 핵심

## Agent Loop란?

Claude Code가 "챗봇"이 아닌 "에이전트"인 이유가 바로 **Agent Loop**다. 일반 챗봇은 질문을 받으면 답변을 생성하고 끝이다. 하지만 Claude Code는 질문을 받으면 **스스로 판단하여 도구를 호출하고, 결과를 확인하고, 필요하면 추가 도구를 호출하는 것을 반복**한다.

이 반복 구조의 핵심은 `query()` async generator 함수다. SSE(Server-Sent Events) 스트리밍으로 응답을 실시간 수신하면서 도구 호출을 감지하고 실행한다.

```
챗봇:
  질문 → 답변 (1회)

에이전트:
  질문 → 도구 호출 → 결과 확인 → 추가 도구 호출 → ... → 최종 답변 (N회)
```

## 노코드 도구에서의 대응 개념

### Agent Loop = Make.com의 시나리오 실행

Make.com에서 시나리오가 실행될 때 조건 분기(Router)에 따라 다른 경로를 탈 수 있는 것처럼, Agent Loop도 매 반복마다 "도구를 쓸지 말지"를 AI가 판단하여 분기한다.

```
Make.com:
  Trigger → Module 1 → Router → [조건A] Module 2 → Module 3
                               → [조건B] Module 4 → 종료

Agent Loop:
  사용자 질문 → API 호출 → [도구 필요] → 실행 → 결과 → API 호출 → ...
                         → [도구 불필요] → 최종 응답
```

### 도구 감지 = Zapier의 "If This Then That" (단, AI가 "If"를 결정)

Zapier에서는 사람이 조건을 설정한다: "Gmail에 이메일이 오면 Slack에 알림을 보내라." Agent Loop에서는 **AI가 조건을 스스로 판단**한다: "파일 구조를 알아야 하니까 Glob을 쓰자."

### Auto-Compact = Gmail의 자동 보관

Gmail에서 받은편지함이 가득 차면 오래된 메일이 자동으로 보관되는 것처럼, Claude Code도 컨텍스트 윈도우가 가득 차면 이전 대화를 **자동으로 요약/압축**한다.

### SSE 스트리밍 = Google Docs의 실시간 편집 표시

Google Docs에서 다른 사람이 타이핑하는 것이 실시간으로 보이는 것처럼, Claude Code도 API 응답이 토큰 단위로 실시간 스트리밍된다. 답변이 완성될 때까지 기다리는 것이 아니라 생성되는 순간부터 화면에 표시된다.

## 실제 동작 원리

### 11단계 Agent Loop 상세

| 단계 | 처리 | 상세 |
|---|---|---|
| 1 | 메시지 수신 | 사용자 입력 또는 도구 실행 결과 |
| 2 | 컨텍스트 조합 | 시스템 프롬프트 + 대화 히스토리 + 도구 결과 |
| 3 | API 요청 | SSE 스트리밍으로 전송 |
| 4 | 토큰 수신 | 실시간으로 토큰 단위 수신 시작 |
| 5 | 도구 호출 감지 | 응답 스트림에서 tool_use 블록 감지 |
| 6 | 권한 확인 | Permission Pipeline으로 실행 가능 여부 판단 |
| 7 | PreToolUse Hook | 등록된 Hook이 있으면 실행 |
| 8 | 도구 실행 | 실제 파일 읽기/쓰기/명령 실행 |
| 9 | PostToolUse Hook | 등록된 Hook이 있으면 실행 |
| 10 | 결과 추가 | 도구 실행 결과를 컨텍스트에 추가 |
| 11 | 반복 판단 | 추가 도구 필요 → 3번으로 / 완료 → 응답 출력 |

### Streaming Tool Executor: 오버랩 실행

일반적인 구현은 "응답 완전 수신 → 도구 실행"이지만, Claude Code는 **응답을 수신하는 중에 도구 실행을 시작**한다. 이것을 Streaming Tool Executor라고 한다.

```
일반적 구현:
  [===응답 수신===] → [도구 실행] → [===응답 수신===] → [도구 실행]

Claude Code:
  [===응답 수신===[도구 실행 시작]===] → [===응답 수신===[도구 실행]===]
                  ↑ 오버랩
```

이 오버랩 덕분에 체감 응답 속도가 빨라진다.

### 도구 동시성 모델: 안전한 도구는 병렬, 위험한 도구는 순차

모든 도구가 같은 방식으로 실행되는 것은 아니다:

| 분류 | 실행 방식 | 최대 동시 수 | 해당 도구 |
|---|---|---|---|
| **Safe (읽기 전용)** | 병렬 실행 | 최대 10개 | Read, Glob, Grep, ListMcpResources |
| **Unsafe (변경 가능)** | 순차 실행 | 1개씩 | Write, Bash, Edit |

```
10개 파일 동시 읽기:
  Read(file1) ─┐
  Read(file2) ─┤
  Read(file3) ─┤
  ...          ├─ 동시 실행 → 전체 결과 반환
  Read(file9) ─┤
  Read(file10)─┘

3개 파일 순차 쓰기:
  Write(file1) → 완료 → Write(file2) → 완료 → Write(file3)
```

이런 설계의 이유: 읽기는 서로 영향을 주지 않지만, 쓰기는 순서가 중요하다 (같은 파일을 동시에 쓰면 충돌).

### Auto-Compact: 컨텍스트가 찰 때의 자동 처리

컨텍스트 윈도우가 가득 차면 Claude Code는 자동으로 압축을 수행한다. 이 과정은 단순하지 않다:

```
Auto-Compact 흐름:

  컨텍스트 사용량 체크
  │
  ├─ 임계값 미만 → 정상 진행
  │
  └─ 임계값 초과 (context_window - 13,000 토큰)
     │
     ▼
     Sub-agent 호출 (요약 전담)
     │
     ├─ 성공 → 이전 대화를 요약본으로 교체
     │         → 50K 이내 파일 내용 복원
     │         → 25K 이내 Skill 내용 재주입
     │         → 대화 계속
     │
     └─ 실패 → Circuit Breaker 카운트 증가
               → 3회 연속 실패 시 압축 중단
               → 사용자에게 새 세션 권고
```

핵심 숫자들:

| 항목 | 값 |
|---|---|
| 압축 트리거 임계값 | context_window - 13,000 토큰 |
| Circuit Breaker 한도 | 연속 3회 실패 |
| 파일 내용 복원 한도 | 50K 토큰 이내 |
| Skill 재주입 한도 | 25K 토큰 이내 |

### 250K/일 API 낭비를 3줄로 해결한 사례

소스 코드에서 발견된 흥미로운 최적화: API 요청 시 불필요한 컨텍스트가 반복적으로 전송되던 문제가 있었다. 이를 **캐시 키 3줄 추가**로 해결하여 하루 약 250K 토큰의 API 호출 낭비를 절감했다.

```
문제:
  매 API 요청마다 시스템 프롬프트가 완전히 새로 전송됨
  → 동일한 내용이 매번 토큰으로 과금

해결 (3줄):
  시스템 프롬프트에 캐시 브레이크포인트 설정
  → API가 이전 요청의 캐시를 재사용
  → 동일 세션 내 반복 전송 제거
```

이것이 오픈 소스 코드 분석의 실질적 가치다 — 프로덕션에서 실제로 일어나는 성능 문제와 해결 방법을 배울 수 있다.

## 실전 시나리오

### "컨텍스트 윈도우가 차고 있다 — 다음에 무슨 일이 일어나는가?"

대화가 길어지고 있을 때의 Auto-Compact 과정:

```
대화 상태:
  ████████████████████████████████░░░░ (약 90% 사용)

1. Agent Loop가 다음 API 요청을 준비
2. 컨텍스트 크기 확인 → 임계값 초과 감지
3. Auto-Compact 트리거

4. Sub-agent 생성 (요약 전담)
   → "이전 대화를 핵심만 남기고 요약해라"
   → Sub-agent가 독립 컨텍스트에서 요약 생성

5. 요약본으로 교체
   이전: [메시지1][메시지2]...[메시지50] = 150K 토큰
   이후: [요약: 핵심 결정사항 5개] = 5K 토큰

6. 중요 파일 복원
   → 최근 읽은 파일 중 50K 이내만 다시 로드
   → 나머지는 "file1.ts를 읽었었음" 수준으로 축약

7. Skill 재주입
   → 활성 Skill이 있었다면 25K 이내로 재주입

8. 대화 계속
   ████████░░░░░░░░░░░░░░░░░░░░░░░░░░ (약 30% 사용)
```

### "Claude가 파일 10개를 동시에 읽는다" — 병렬 배칭 설명

대규모 코드 분석을 요청했을 때:

```
사용자: "src/ 디렉토리의 모든 TypeScript 파일을 읽고 아키텍처를 분석해줘"

1. Glob 실행 → 12개 .ts 파일 발견

2. Claude가 Read 도구를 12회 호출하기로 결정

3. Tool Executor가 분류:
   Read = Safe (읽기 전용) → 병렬 실행 가능

4. 배치 실행:
   Batch 1: Read(file1~file10)  → 10개 동시 실행 (최대 동시 수)
   Batch 2: Read(file11~file12) → 2개 동시 실행

5. 결과: 12개 파일을 2라운드에 읽기 완료
   순차 실행이었다면: 12라운드 필요
   병렬 실행 결과: 2라운드로 단축 (6배 빠름)
```

이 병렬 실행은 사용자가 신경 쓸 필요 없다 — Claude Code가 도구의 안전성을 자동으로 판단하여 최적의 실행 방식을 선택한다.

## 핵심 정리

- Agent Loop = Claude Code를 **에이전트**로 만드는 핵심 반복 구조
- `query()` async generator로 SSE 스트리밍 + 도구 감지 + 실행을 하나의 흐름으로 처리
- **Streaming Tool Executor**: 응답 수신 중 도구 실행을 오버랩하여 체감 속도 향상
- **도구 동시성**: Safe 도구는 최대 10개 병렬, Unsafe 도구는 순차 실행
- **Auto-Compact**: 임계값 초과 시 Sub-agent 요약, 50K 파일 복원, 25K Skill 재주입, Circuit Breaker 3회
- 캐시 최적화 3줄로 250K/일 API 낭비 절감 — 프로덕션 코드에서 배우는 실전 최적화

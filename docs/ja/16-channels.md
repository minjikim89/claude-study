# チャンネル — 外部システムがClaudeに話しかける仕組み

## チャンネルとは？

これまでのClaude Codeは常に**自分が先に話しかけて**反応していました。ターミナルの前に座ってメッセージを送り、Claudeが答えるという構造です。

チャンネルはこの方向を逆転させます。**外部システム（Telegram、Discord、CIパイプラインなど）がClaudeにイベントをpushし、Claudeがそれを処理して同じチャンネルに返信**する仕組みです。

ターミナルの前にいなくても構いません。Telegramのメッセージが1件届くと、自分のコンピューターで実行中のClaudeがそれを読み、ローカルファイルを開き、作業して、再びTelegramに結果を送ります。

> まとめ: チャンネル = **外部世界がClaudeに話しかける窓口**

## ノーコードツールとの比較

| ノーコードツール | チャンネルの対応 |
|---|---|
| **ZapierのWebhook Trigger** | 外部イベントが届いたらワークフローを開始 |
| **MakeのWebhookモジュール** | CIイベント、監視アラートを受け取って処理 |
| **Slackボット** | チャットにメッセージを送るとボットが応答 |
| **Discordボット** | サーバー/DMでボットと双方向の会話 |
| **n8nのWebhook Node** | HTTPリクエストを受けて自動化を開始 |

主な違い:
- ノーコードボットは**クラウドで実行** → ローカルファイルにアクセス不可
- チャンネルは**自分のコンピューターのClaudeが処理** → ローカルファイル、Git、MCPサーバーすべて使用可能
- ノーコードは**設定済みの自動化のみ**実行 → チャンネルは**Claudeが状況を判断**して処理

## 仕組み

```
┌─ 外部 ───────────────────────────────────────────────┐
│                                                      │
│  Telegram                Discord               CI    │
│   スマホのメッセージ        DM / サーバー          アラート│
│      │                      │                   │    │
└──────┼──────────────────────┼───────────────────┼────┘
       │                      │                   │
       ▼                      ▼                   ▼
┌─ チャンネルプラグイン（MCPサーバー）────────────────────┐
│                                                        │
│  メッセージ/イベントを受信 → <channel>イベントに変換     │
│                                                        │
└────────────────────────┬───────────────────────────────┘
                         │ push
                         ▼
┌─ 自分のコンピューター — Claude Codeセッション（実行中）────┐
│                                                        │
│  <channel source="telegram"> イベントを受信             │
│  → Claudeが読んで判断                                   │
│  → ローカルファイル/Git/MCPサーバーで作業                 │
│  → replyツールで同じチャンネルに返信                     │
│                                                        │
└────────────────────────────────────────────────────────┘
```

**核心の構造**:
- チャンネルプラグインは**MCPサーバーの特殊な形態** — ただし通常のMCPと違い、ClaudeがpullするのではなくイベントがClaudeに**push**される
- イベントは**セッションが開いている間のみ**届く（バックグラウンドで維持する必要がある）
- ターミナル出力: 受信メッセージは表示されますが、**返信内容はターミナルには表示されずチャンネルに直接送信**されます

## チャンネル vs リモートコントロール vs Scheduled Tasks

3つの機能はどれも「ターミナルの外からClaudeを扱う」印象がありますが、**誰が主体か**がまったく異なります。

| | **リモートコントロール** | **チャンネル** | **Scheduled Tasks** |
|---|---|---|---|
| **主体** | 自分がセッションをモバイルから直接操作 | 外部システムがClaudeにイベントをpush | 時間になるとClaudeが自動実行 |
| **トリガー** | 自分が直接メッセージを入力 | 外部イベントの到着（メッセージ、アラート、webhook） | 設定された時間/サイクル |
| **フローの方向** | 自分 → Claude | 外部 → Claude → 外部 | Claudeが自動実行 |
| **適した状況** | 進行中の作業を引き続き操作 | CIアラートの処理、ボットの応答、リモート指示 | 定期レポート、定期的な監視 |
| **例え** | テレビのリモコン | 呼び鈴 | 目覚まし時計 |

> **簡単な判断基準**:
> 「スマホから自分でコマンドしたい」→ リモートコントロール
> 「外部システムがClaudeに通知を送って処理させたい」→ チャンネル
> 「時間になったら自動で動かしておきたい」→ Scheduled Tasks

## チャンネル vs 通常のMCP

同じプラグイン/MCPの仕組みを使いますが、動作方式が異なります。

| | **通常のMCPサーバー** | **チャンネル** |
|---|---|---|
| **方向** | Claudeが必要なときにpull | 外部がClaudeにpush |
| **トリガー** | Claudeの作業中に「DBからデータを取得して」 | 外部メッセージ/イベントの到着 |
| **例** | Notion MCP、Slack MCP（参照用） | Telegramチャンネル、Discordチャンネル |
| **例え** | 図書館の司書（聞けば出してくれる） | 郵便配達員（手紙が届いたら呼び鈴を押す） |

## サポートされているチャンネル（リサーチプレビュー）

現在の公式サポートチャンネル:

| チャンネル | プラグイン名 | 用途 |
|---|---|---|
| **Telegram** | `telegram@claude-plugins-official` | ボットDM、グループメッセージ |
| **Discord** | `discord@claude-plugins-official` | DM、サーバーチャンネル |
| **fakechat** | `fakechat@claude-plugins-official` | localhostデモ（テスト用） |

> `fakechat` はブラウザ（`http://localhost:8787`）からすぐにテストできる公式デモチャンネルです。実際のアプリアカウントなしにチャンネルの動作を体験するときに便利です。

## インストールと設定 — Telegram基準

### 事前準備

```bash
# Bunが必要（チャンネルプラグインはBunスクリプト）
bun --version        # なければインストールが必要
```

Bunのインストール: https://bun.sh/docs/installation

### ステップ1: プラグインのインストール

```
/plugin install telegram@claude-plugins-official
```

初めての場合はMarketplaceの登録が先:
```
/plugin marketplace add anthropics/claude-plugins-official
```

### ステップ2: ボットトークンの設定

[BotFather](https://t.me/BotFather)で `/newbot` → トークンをコピーして:

```
/telegram:configure <トークン>
```

→ `~/.claude/channels/telegram/.env` に保存されます

### ステップ3: チャンネルを有効化して実行

```bash
claude --channels plugin:telegram@claude-plugins-official
```

### ステップ4: ペアリング（初回のみ1回）

1. Telegramで自分のボットに何かメッセージを送る
2. ボットがペアリングコードを返す
3. Claude Codeで入力:

```
/telegram:access pair <コード>
/telegram:access policy allowlist    ← 自分のアカウントのみ許可（セキュリティ）
```

### Discordの場合は？

構造は同じです。[Discord開発者ポータル](https://discord.com/developers/applications)でボットを作成 → Message Content Intentを有効化 → サーバーに招待 → トークンを設定します。

```
/plugin install discord@claude-plugins-official
/discord:configure <トークン>
claude --channels plugin:discord@claude-plugins-official
```

## セキュリティの仕組み

すべてのチャンネルプラグインは**送信者の許可リスト（allowlist）**を維持しています。

```
許可リストにないアカウントがメッセージを送った場合
        → 静かに無視される（エラーなし）

許可リストにあるアカウントがメッセージを送った場合
        → Claude Codeにchannelイベントとして送信
```

ペアリング後に `/telegram:access policy allowlist` を設定すると、**登録されたアカウントのみ**Claudeにアクセスできるようになります。

さらに:
- チャンネルが有効になるには `--channels` フラグが必ず必要（`.mcp.json` のみではメッセージを受信できない）
- Team/EnterpriseプランはAdmin が `channelsEnabled` の設定をオンにする必要がある

## 実践シナリオ

### 1. スマートフォンからのリモート指示
```
【状況】外出中。自宅のコンピューターでClaude Codeが動いている

Telegramを送信: "今日変更したファイルの一覧とGitの状態を教えて"

→ 自分のコンピューターのClaudeがGitコマンドを実行
→ 結果をTelegramで返信
→ スマホですぐに確認
```

### 2. CIの失敗アラート → 即時分析
```
【状況】Claude Codeのセッションを開いたままバックグラウンドでCIを実行中

Webhook → チャンネル: "ビルド失敗: TypeError in auth.ts line 42"

→ Claudeが該当ファイルを開いてエラーを分析
→ Discordで "auth.tsの42行目でundefined参照。修正案をコミットしますか？" と返信
→ スマホで "はい" と送るとClaudeが実際にファイルを修正 + コミット
```

### 3. 長時間作業の監視 + 介入
```
【状況】大規模なリファクタリングを開始、1〜2時間かかる見込み

claude --channels plugin:telegram@claude-plugins-official で実行

途中でTelegram: "今どこまで進んだ？"
→ Claude: "src/api/ 完了、src/db/ 作業中（43%）"

Telegram: "db/legacy.tsは触らないで"
→ Claudeがすぐに方向を修正
```

## 使用条件

| 項目 | 条件 |
|---|---|
| Claude Codeバージョン | v2.1.80以上 |
| 認証方法 | claude.aiでのログイン（コンソール/APIキーは未サポート） |
| 依存関係 | Bunのインストールが必要 |
| プラン | Pro/Max（個人）、Enterpriseは管理者による有効化が必要 |
| 状態 | リサーチプレビュー（--channelsフラグによるopt-in方式） |

## まとめ

- チャンネル = **外部イベントが実行中のセッションにpush**される仕組み（MCPのpushバージョン）
- Telegram、Discordのボットを接続すれば**チャット画面がClaudeの入出力窓口**になる
- 自分のコンピューターで実行 → **ローカルファイル、Git、MCPサーバーをすべて活用可能**
- リモートコントロール（自分が操作）と違い、チャンネルは**外部システムがClaudeに話しかける**仕組み
- セキュリティ: 許可リストベース — 登録されていないアカウントのメッセージは無視
- `--channels` フラグでセッションごとにopt-in、`Bun` が必須、**v2.1.80+** が必要
- リサーチプレビュー段階 — `--channels` フラグの文法とプロトコルは変更される可能性あり

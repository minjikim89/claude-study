# Agent Loopの解剖 — エージェントをエージェントたらしめるコア

## Agent Loopとは？

Claude Codeが「チャットボット」ではなく「エージェント」である理由が、まさに**Agent Loop**です。通常のチャットボットは質問を受けて回答を生成して終わりです。しかしClaude Codeは質問を受けると、**自ら判断してツールを呼び出し、結果を確認し、必要に応じて追加のツールを呼び出すことを繰り返します**。

この繰り返し構造の核心は`query()` async generator関数です。SSE（Server-Sent Events）ストリーミングでリアルタイムにレスポンスを受信しながら、ツール呼び出しを検出して実行します。

```
チャットボット:
  質問 → 回答（1回）

エージェント:
  質問 → ツール呼び出し → 結果確認 → 追加ツール呼び出し → ... → 最終回答（N回）
```

## ノーコードツールでの対応概念

### Agent Loop = Make.comのシナリオ実行

Make.comでシナリオが実行されるとき、条件分岐（Router）に応じて異なるパスを取れるように、Agent Loopも毎回の繰り返しで「ツールを使うか使わないか」をAIが判断して分岐します。

```
Make.com:
  Trigger → Module 1 → Router → [条件A] Module 2 → Module 3
                               → [条件B] Module 4 → 終了

Agent Loop:
  ユーザー質問 → API呼び出し → [ツール必要] → 実行 → 結果 → API呼び出し → ...
                              → [ツール不要] → 最終応答
```

### ツール検出 = Zapierの「If This Then That」（ただし、AIが「If」を決める）

Zapierでは人間が条件を設定します：「Gmailにメールが来たらSlackに通知を送る。」Agent Loopでは**AIが条件を自ら判断します**：「ファイル構造を知る必要があるからGlobを使おう。」

### Auto-Compact = Gmailの自動アーカイブ

Gmailで受信トレイがいっぱいになると古いメールが自動的にアーカイブされるように、Claude Codeもコンテキストウィンドウがいっぱいになると以前の会話を**自動的に要約/圧縮**します。

### SSEストリーミング = Google Docsのリアルタイム編集表示

Google Docsで他の人がタイピングしているのがリアルタイムで見えるように、Claude CodeもAPIレスポンスをトークン単位でリアルタイムにストリーミングします。回答が完成するまで待つのではなく、生成される瞬間から画面に表示されます。

## 実際の動作原理

### 11ステップAgent Loopの詳細

| ステップ | 処理 | 詳細 |
|---|---|---|
| 1 | メッセージ受信 | ユーザー入力またはツール実行結果 |
| 2 | コンテキスト組み立て | システムプロンプト + 会話履歴 + ツール結果 |
| 3 | APIリクエスト | SSEストリーミングで送信 |
| 4 | トークン受信 | リアルタイムでトークン単位の受信開始 |
| 5 | ツール呼び出し検出 | レスポンスストリームでtool_useブロックを検出 |
| 6 | 権限確認 | Permission Pipelineで実行可否を判断 |
| 7 | PreToolUse Hook | 登録されたHookがあれば実行 |
| 8 | ツール実行 | 実際のファイル読み書き/コマンド実行 |
| 9 | PostToolUse Hook | 登録されたHookがあれば実行 |
| 10 | 結果追加 | ツール実行結果をコンテキストに追加 |
| 11 | ループ判断 | 追加ツール必要 → ステップ3へ / 完了 → レスポンス出力 |

### Streaming Tool Executor：オーバーラップ実行

一般的な実装は「レスポンス完全受信 → ツール実行」ですが、Claude Codeは**レスポンスを受信している最中にツール実行を開始**します。これをStreaming Tool Executorと呼びます。

```
一般的な実装:
  [===レスポンス受信===] → [ツール実行] → [===レスポンス受信===] → [ツール実行]

Claude Code:
  [===レスポンス受信===[ツール実行開始]===] → [===レスポンス受信===[ツール実行]===]
                       ↑ オーバーラップ
```

このオーバーラップにより、体感レスポンス速度が大幅に向上します。

### ツール並行性モデル：安全なツールは並列、危険なツールは順次

すべてのツールが同じ方法で実行されるわけではありません：

| 分類 | 実行方式 | 最大同時数 | 対象ツール |
|---|---|---|---|
| **Safe（読み取り専用）** | 並列実行 | 最大10個 | Read、Glob、Grep、ListMcpResources |
| **Unsafe（変更可能）** | 順次実行 | 1つずつ | Write、Bash、Edit |

```
10ファイル同時読み込み:
  Read(file1) ─┐
  Read(file2) ─┤
  Read(file3) ─┤
  ...          ├─ 同時実行 → 全結果を返却
  Read(file9) ─┤
  Read(file10)─┘

3ファイル順次書き込み:
  Write(file1) → 完了 → Write(file2) → 完了 → Write(file3)
```

この設計の理由：読み込みは互いに影響しませんが、書き込みは順序が重要です（同じファイルに同時に書き込むと競合が発生します）。

### Auto-Compact：コンテキストが埋まったときの自動処理

コンテキストウィンドウが埋まると、Claude Codeは自動的に圧縮を実行します。このプロセスは単純ではありません：

```
Auto-Compactフロー:

  コンテキスト使用量チェック
  │
  ├─ 閾値未満 → 通常進行
  │
  └─ 閾値超過（context_window - 13,000トークン）
     │
     ▼
     Sub-agent呼び出し（要約専任）
     │
     ├─ 成功 → 以前の会話を要約に置き換え
     │         → 50K以内のファイル内容を復元
     │         → 25K以内のSkill内容を再注入
     │         → 会話を継続
     │
     └─ 失敗 → Circuit Breakerカウント増加
               → 3回連続失敗で圧縮中断
               → ユーザーに新しいセッションを推奨
```

キーナンバー：

| 項目 | 値 |
|---|---|
| 圧縮トリガー閾値 | context_window - 13,000トークン |
| Circuit Breaker上限 | 連続3回失敗 |
| ファイル内容復元上限 | 50Kトークン以内 |
| Skill再注入上限 | 25Kトークン以内 |

### 250K/日のAPI無駄遣いを3行で解決した事例

ソースコードで見つかった興味深い最適化：APIリクエスト時に不要なコンテキストが繰り返し送信されていた問題がありました。**キャッシュキー3行の追加**でこれを解決し、1日あたり約250Kトークンの無駄なAPI呼び出しを削減しました。

```
問題:
  毎回のAPIリクエストでシステムプロンプトが完全に新規送信
  → 同一内容が毎回トークンとして課金

解決（3行）:
  システムプロンプトにキャッシュブレークポイントを設定
  → APIが前回リクエストのキャッシュを再利用
  → 同一セッション内の重複送信を排除
```

これがオープンソースコード分析の実質的な価値です — プロダクション環境で実際に発生するパフォーマンス問題とその解決方法を学べます。

## 実践シナリオ

### 「コンテキストウィンドウが埋まってきた — 次に何が起こるのか？」

会話が長くなっているときのAuto-Compactプロセス：

```
会話状態:
  ████████████████████████████████░░░░ （約90%使用）

1. Agent Loopが次のAPIリクエストを準備
2. コンテキストサイズ確認 → 閾値超過を検知
3. Auto-Compactトリガー

4. Sub-agent生成（要約専任）
   → "以前の会話を核心だけ残して要約せよ"
   → Sub-agentが独立コンテキストで要約を生成

5. 要約に置き換え
   以前: [メッセージ1][メッセージ2]...[メッセージ50] = 150Kトークン
   以後: [要約: 主要決定事項5件] = 5Kトークン

6. 重要ファイルの復元
   → 最近読んだファイルのうち50K以内のみ再ロード
   → 残りは「file1.tsを読んだ」レベルに縮約

7. Skillの再注入
   → アクティブなSkillがあれば25K以内で再注入

8. 会話を継続
   ████████░░░░░░░░░░░░░░░░░░░░░░░░░░ （約30%使用）
```

### 「Claudeがファイル10個を同時に読む」 — 並列バッチングの説明

大規模コード分析をリクエストしたとき：

```
ユーザー: "src/ディレクトリのすべてのTypeScriptファイルを読んでアーキテクチャを分析して"

1. Glob実行 → 12個の.tsファイルを発見

2. ClaudeがReadツールを12回呼び出すことを決定

3. Tool Executorが分類:
   Read = Safe（読み取り専用）→ 並列実行可能

4. バッチ実行:
   Batch 1: Read(file1~file10)  → 10個同時実行（最大同時数）
   Batch 2: Read(file11~file12) → 2個同時実行

5. 結果: 12ファイルを2ラウンドで読み込み完了
   順次実行の場合: 12ラウンド必要
   並列実行の結果: 2ラウンドに短縮（6倍高速）
```

この並列実行はユーザーが気にする必要はありません — Claude Codeがツールの安全性を自動的に判断し、最適な実行方式を選択します。

## まとめ

- Agent Loop = Claude Codeを**エージェント**にするコアの繰り返し構造
- `query()` async generatorでSSEストリーミング + ツール検出 + 実行を一つのフローで処理
- **Streaming Tool Executor**：レスポンス受信中にツール実行をオーバーラップして体感速度を向上
- **ツール並行性**：Safeツールは最大10個並列、Unsafeツールは順次実行
- **Auto-Compact**：閾値超過時にSub-agent要約、50Kファイル復元、25K Skill再注入、Circuit Breaker 3回
- キャッシュ最適化3行で250K/日のAPI無駄遣いを削減 — プロダクションコードから学ぶ実践的最適化

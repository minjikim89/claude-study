# Agent Teams — チームメンバーが直接コミュニケーションするプロジェクトチーム

## Agent Teams とは？

Subagent が**「部下がメインにだけ報告する構造」**なら、Agent Teams は**「チームメンバーが互いに直接コミュニケーションしながら協働する構造」**です。

複数の Claude インスタンスがそれぞれ独立したセッションで同時に作業しながら、共有タスクリストを通じて進捗を調整します。

> 現在 Agent Teams は**実験的機能（Experimental）**であり、別途設定で有効化する必要があります。

## Subagent と Agent Teams の決定的な違い

```
Subagent の構造：
┌─ メイン ─────────────────┐
│                          │
│  Subagent A ──▶ メイン   │   A はメインにのみ報告
│  Subagent B ──▶ メイン   │   B はメインにのみ報告
│  Subagent C ──▶ メイン   │   A、B、C 同士の通信不可
│                          │
└──────────────────────────┘

Agent Teams の構造：
┌─ リーダー ─────────────────────────────────────┐
│                                                │
│  ┌─ Agent A ─┐  ┌─ Agent B ─┐  ┌─ Agent C ─┐ │
│  │ 市場調査  │  │ レポート  │  │ 発表資料  │ │
│  └─────┬─────┘  └─────┬─────┘  └─────┬─────┘ │
│        │              │              │        │
│        └──────── メッセージ ──────────┘        │
│                    +                           │
│            共有タスクリスト                      │
└────────────────────────────────────────────────┘
```

## 詳細比較表

| | **Subagent** | **Agent Teams** |
|---|---|---|
| **例え** | 部下（1:1 報告） | プロジェクトチーム（水平コミュニケーション） |
| **通信方式** | メインにのみ結果を返却 | チームメンバー間でメッセージ ＋ 共有タスクリスト |
| **調整方式** | メインがすべて管理・調整 | タスクリストで自律的に調整 |
| **コンテキスト** | 独立（結果のみ返却） | 各自が完全に独立したセッション |
| **並列性** | 限定的（メインが順次管理） | 高い（同時実行） |
| **トークンコスト** | 低い（1 つずつ実行） | 高い（N セッション同時） |
| **有効化** | デフォルトで使用可能 | 設定が必要（実験的） |
| **適した作業** | 単一作業の委任 | 複数作業の同時協働 |

## ノーコードツールでの例え

| ノーコードツールの概念 | Claude Code |
|---|---|
| Make の単一シナリオで HTTP モジュールを呼び出す | Subagent |
| Make の複数シナリオが Webhook で連携 | Agent Teams |
| Notion で一人が複数ページを作業 | Subagent |
| Notion で複数人がそれぞれのページを同時に作成 | Agent Teams |
| Slack でマネージャーが 1:1 で業務指示 | Subagent |
| Slack チャンネルでチームメンバーが直接コミュニケーションしながら協働 | Agent Teams |

## Agent Teams の有効化方法

`~/.claude/settings.json` ファイルに以下を追加：

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

設定後に Claude Code を再起動すると使用可能になります。

## Agent Teams の仕組み

### 1. リーダー＆メンバー構造

```
ユーザー：「フロントエンド、バックエンド、テストを同時に作業して」
  │
  ▼
リーダー Claude（メインセッション）
  │
  ├─▶ Agent A：フロントエンド開発（独立セッション）
  ├─▶ Agent B：バックエンド API 開発（独立セッション）
  └─▶ Agent C：テストコード作成（独立セッション）
```

### 2. 共有タスクリスト

チームメンバーは**共有タスクリスト**を通じて互いの進捗を把握します：

```
[共有タスクリスト]
├── ✅ Agent A：コンポーネント構造設計 完了
├── 🔄 Agent B：API エンドポイント実装 進行中
├── ⏳ Agent C：A、B の完了待ち（依存関係）
└── 📋 未割り当て：統合テスト作成
```

### 3. チームメンバー間の直接メッセージ

Subagent と異なり、Agent Teams のメンバーは**互いに直接メッセージ**を送ることができます：

```
Agent A（フロント）：「API のレスポンス形式は { data: [] } ですか？」
Agent B（バックエンド）：「はい、エラー時は { error: message } 形式です」
Agent C（テスト）：「両方の形式をテストケースに反映します」
```

## 使い分けシナリオ

### Subagent が適している場合

- **単一作業の委任**：「この PDF を要約して」「コードベースの構造を探索して」
- **順次パイプライン**：A 完了 → B 開始 → C 開始
- **コスト効率重視**：トークン使用量を最小化しなければならないとき
- **シンプルな分析・報告**：結果だけ必要で過程は重要でないとき

### Agent Teams が適している場合

- **並列開発**：「フロント / バック / テストを同時に開発」
- **多角的レビュー**：「3 人の専門家が同時に PR レビュー」
- **相互依存作業**：メンバー A の結果がメンバー B に必要で、B の結果が C に必要な場合
- **大規模リファクタリング**：複数のモジュールを同時に修正しなければならないとき

## 実践例

### 例 1：PR レビュー（Agent Teams）

```
リーダー：「この PR を 3 つの観点でレビューして」
  │
  ├─▶ セキュリティ専門家：SQL インジェクション、XSS などを検討
  ├─▶ パフォーマンス専門家：N+1 クエリ、メモリリークなどを検討
  └─▶ アーキテクチャ専門家：パターンの一貫性、結合度などを検討
  │
  ▼
リーダー：3 つのレビューを統合して最終フィードバックを作成
```

### 例 2：フルスタック機能開発（Agent Teams）

```
リーダー：「ユーザープロフィール編集機能を追加して」
  │
  ├─▶ Agent A：React コンポーネント ＋ フォーム UI 開発
  ├─▶ Agent B：Express API エンドポイント開発
  └─▶ Agent C：Jest ＋ Cypress テスト作成
  │
  ▼
[A から B へ]：「プロフィール画像アップロード API も必要です」
[B から C へ]：「アップロード API エンドポイント /api/profile/image を作りました」
[C]：「アップロード API のテストも追加します」
```

## コストとトレードオフ

| 項目 | Subagent | Agent Teams |
|---|---|---|
| **トークン使用量** | 少ない（一度に 1 つ） | 多い（N 個同時） |
| **所要時間** | 順次処理なので遅い可能性あり | 並列処理なので速い |
| **複雑さ** | シンプル | 調整のオーバーヘッドあり |
| **結果の品質** | 単一の視点 | 多角的な視点の統合 |

> **一般原則**：作業が独立していてシンプルなら Subagent、作業が相互依存していて複雑なら Agent Teams。

## まとめ

- Agent Teams = **複数の Claude インスタンスが独立セッションで同時作業 ＋ 直接通信**
- Subagent との核心的な違い：メンバー間の**直接メッセージ** ＋ **共有タスクリスト**
- 現在実験的機能 → `settings.json` で有効化が必要
- コスト（トークン）は高いが、並列処理によってスピードと品質を向上できる
- 実務ではほとんど Subagent で十分で、大規模・複雑な作業で Agent Teams を検討

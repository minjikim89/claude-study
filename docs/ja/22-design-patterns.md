# 設計パターン12選 — プロダクションコードから抽出したエージェントパターン

## なぜ設計パターンなのか？

Claude Codeのソースコードから**8つのコアパターン + 4つのインフラパターン**を抽出しました。これらは単なるコード技法ではなく、エージェントシステムが安定的に動作するための**必須設計戦略**です。

一般的なアプリケーションとエージェントシステムの決定的な違い：エージェントは**次に何をするか予測不可能**です。ユーザーの入力に応じてファイルを読むかもしれないし、コマンドを実行するかもしれないし、何もしないかもしれません。この不確実性の中で安定性を維持するには、堅牢なパターンが必要です。

## ノーコードツールでの対応概念

12個のパターンそれぞれを身近なツールでたとえると：

### コアパターン8個

| # | パターン | ノーコードでのたとえ |
|---|---|---|
| 1 | **Generator Streaming** | Google Docsで相手がタイピングしているのをリアルタイムで見ること |
| 2 | **Feature Gate** | NotionのFeature Previewトグル — 機能のオン/オフが可能 |
| 3 | **Memoized Context** | ブラウザがよく訪れるページをキャッシュすること |
| 4 | **Withhold & Recover** | オートコレクトがタイプミスに気づく前に修正すること |
| 5 | **Lazy Import** | アプリが現在の画面だけロードし、残りは遷移時にロードすること |
| 6 | **Immutable State** | Google Sheetsのバージョン履歴 — 誤って上書きできない |
| 7 | **Interruption Resilience** | Google Docsの自動保存 — ブラウザがクラッシュしても内容が保持される |
| 8 | **Dependency Injection** | USBポート — 同じポートに異なるデバイスを接続できる |

### インフラパターン4個

| # | パターン | ノーコードでのたとえ |
|---|---|---|
| 9 | **Tool Concurrency** | 料理における並列 vs 順次作業 — 野菜の下ごしらえは同時に、炒めるのは順番に |
| 10 | **Auto-Compact** | 会議ノートがいっぱいになったとき要点だけ要約すること |
| 11 | **Permission Pipeline** | ビルのセキュリティ — ロビー、エレベーター、オフィスのドアそれぞれでチェック |
| 12 | **API Retry** | 電話が切れたときの自動リダイヤル |

## 実際の動作原理

### パターン1：Generator Streaming

**何か**：APIレスポンスを一括で受け取るのではなく、トークン単位でリアルタイムにストリーミング表示。

```
従来の方式:
  [====待機====] → [レスポンス全体を一度に表示]

Generator Streaming:
  [ト][ー][ク][ン][ず][つ] → [リアルタイムに画面に表示]
```

**Claude Code実装**：`query()`関数がasync generatorとして実装され、`yield`で各トークンを返却。React Inkがリアルタイムにレンダリング。

### パターン2：Feature Gate

**何か**：ランタイムに機能をオン/オフできるトグルシステム。

```
Feature Gateチェック:
  機能Xの使用リクエスト
  │
  ├─ Gate ON  → 機能を実行
  └─ Gate OFF → 代替パスまたは無効化
```

**Claude Code実装**：7つの実行モード、Betaフィーチャートグル、MCPサーバー別のツール有効化/無効化がすべてFeature Gateパターン。

### パターン3：Memoized Context

**何か**：一度計算した結果をキャッシュして繰り返し計算を防止。

```
初回呼び出し:
  CLAUDE.mdパース → 結果生成 → キャッシュに保存 → 返却

2回目の呼び出し:
  キャッシュ確認 → 結果あり → 直接返却（パース省略）
```

**Claude Code実装**：システムプロンプト、CLAUDE.md、環境情報がセッション内でキャッシュ。14個のキャッシュベクターが存在。

### パターン4：Withhold & Recover

**何か**：問題を事前に検知してユーザーが気づく前に復旧。

```
一般的なエラー処理:
  実行 → エラー発生 → ユーザーにエラー表示 → 手動復旧

Withhold & Recover:
  実行 → エラー検知 → 自動リトライ/復旧 → ユーザーは気づかない
```

**Claude Code実装**：ツール実行失敗時の自動リトライ、コンテキスト破損時の自動復旧、APIエラー時のバックオフリトライ。

### パターン5：Lazy Import

**何か**：すべてのモジュールを起動時にロードせず、必要なときだけロード。

```
Eager Import（起動が遅い）:
  起動 → [モジュールAロード][モジュールBロード][モジュールCロード] → 準備完了

Lazy Import（起動が速い）:
  起動 → [コアのみロード] → 準備完了
  ... 後で必要になったとき → [モジュールBロード]
```

**Claude Code実装**：MCPサーバー接続、Pluginロード、重いユーティリティがすべてLazy Import。888KBの初期バンドルサイズを最小化。

### パターン6：Immutable State

**何か**：状態を直接変更せず、新しい状態を生成して置き換え。

```
Mutable（危険）:
  state.messages.push(newMsg)  → 以前の状態が消える

Immutable（安全）:
  newState = {...state, messages: [...state.messages, newMsg]}
  → 以前の状態が保持される
```

**Claude Code実装**：Zustandストアで会話履歴をImmutableに管理。以前の状態にロールバック可能。

### パターン7：Interruption Resilience

**何か**：作業中に中断されてもデータが失われない設計。

```
中断シナリオ:
  ファイル書き込み中 → Ctrl+C → ？

脆弱な実装: ファイルが半分だけ書かれて破損
Resilientな実装: 一時ファイルに書き込み → 完了後に置き換え（atomic write）
```

**Claude Code実装**：Memoryの自動保存、セッション状態の復旧、ツール実行中のインタラプト処理。

### パターン8：Dependency Injection（DI）

**何か**：具体的な実装ではなくインターフェースに依存し、交換可能にすること。

```
DIなし:
  Agent Loop → 直接Anthropic APIを呼び出し

DIあり:
  Agent Loop → API Interface → [Anthropic API / Mock API / Local Model]
  同じコードが異なる環境で動作可能
```

**Claude Code実装**：APIクライアント、ファイルシステムアクセス、権限チェッカーがすべてDIパターン。テスト時にMockに交換可能。

### パターン9：Tool Concurrency

**何か**：安全なツールは並列で、危険なツールは順次で実行する戦略。

```
Safeツール（Read、Glob）：最大10個並列実行
Unsafeツール（Write、Bash）：1つずつ順次実行
```

### パターン10：Auto-Compact

**何か**：コンテキスト限界に達するとSub-agentが自動的に要約/圧縮。

```
閾値: context_window - 13,000トークン
失敗時: Circuit Breaker 3回で中断
復元: ファイル50K + Skill 25K再注入
```

### パターン11：Permission Pipeline

**何か**：ツール実行前に複数段階の権限確認を通過する必要がある構造。

```
ツール実行リクエスト
  │
  ▼
  ステージ1: ツールタイプ確認（Safe/Unsafe）
  │
  ▼
  ステージ2: プロジェクト設定確認（allowlist/denylist）
  │
  ▼
  ステージ3: ユーザー設定確認（自動許可の有無）
  │
  ▼
  ステージ4: ユーザーに確認を要求（必要な場合）
  │
  ▼
  実行許可または拒否
```

**Claude Code実装**：23ステップの権限チェックパイプラインがすべてのツール実行前に動作。

### パターン12：API Retry

**何か**：API呼び出し失敗時に指数バックオフ（exponential backoff）でリトライ。

```
試行1: 失敗 → 1秒待機
試行2: 失敗 → 2秒待機
試行3: 失敗 → 4秒待機
試行4: 成功 → 結果返却

最大リトライ: 通常3〜5回
```

## 実践シナリオ

### 「自分のプロジェクトにはどのパターンが必要か？」

プロジェクトタイプによって優先適用すべきパターンが異なります：

**Webアプリ（Next.js、Reactなど）**

| 優先度 | パターン | 理由 |
|---|---|---|
| 高 | Lazy Import | バンドルサイズがロード速度に直接影響 |
| 高 | Immutable State | Reactの状態管理の基本原則 |
| 中 | Feature Gate | A/Bテスト、段階的な機能デプロイに必須 |
| 中 | API Retry | ネットワーク不安定環境への対策 |

**自動化スクリプト（Python、Node.js）**

| 優先度 | パターン | 理由 |
|---|---|---|
| 高 | API Retry | 外部API依存度が高いスクリプトの安定性 |
| 高 | Interruption Resilience | 長時間タスクの中断時のデータ保護 |
| 中 | Withhold & Recover | エラーが出ても自動復旧して無人実行が可能 |

**AIエージェントシステム**

| 優先度 | パターン | 理由 |
|---|---|---|
| 高 | Generator Streaming | ユーザーの体感速度向上 |
| 高 | Tool Concurrency | エージェント実行効率の最大化 |
| 高 | Auto-Compact | 長い会話の安定的な維持 |
| 高 | Permission Pipeline | エージェントの行動範囲の制御 |
| 中 | Memoized Context | 繰り返し計算の排除でコスト削減 |
| 中 | DI | 様々なモデル/ツールの交換をサポート |

### パターン組み合わせのシナジー

12個のパターンは単独でも有用ですが、組み合わせるとシナジーが生まれます：

```
例: "ユーザーが大規模コード分析をリクエストしたとき"

1. Generator Streaming → レスポンスをリアルタイムに表示
2. Tool Concurrency → 10ファイルを並列で読み込み
3. Memoized Context → すでにパース済みのCLAUDE.mdをキャッシュから再利用
4. Permission Pipeline → 各ファイル読み込み前に権限確認
5. Auto-Compact → コンテキストが埋まったら自動要約
6. API Retry → 途中でAPIエラーが出たら自動リトライ
```

6つのパターンが1つのリクエストを処理するのに同時に動作します。これがエージェントシステムが単純なAPIラッパーと異なる点です。

## まとめ

- Claude Codeのソースから**8つのコア + 4つのインフラ = 12の設計パターン**を抽出
- エージェントシステムの核心的課題：予測不可能な実行フローでの安定性確保
- キーナンバー：250K/日削減、23ステップ権限、14キャッシュベクター、888KBバンドル、最大10並列
- プロジェクトタイプ（Webアプリ / 自動化 / AIエージェント）に応じて優先パターンが異なる
- 12のパターンは組み合わさってシナジーを発揮 — 1つのリクエストに複数のパターンが同時に動作

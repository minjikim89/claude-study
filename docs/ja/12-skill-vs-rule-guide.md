# スキル vs ルール — いつ何を使うか判断する方法

## なぜこの区別が重要か

Claude Codeで「Claudeに指示を出す方法」は大きく4つあります:

```
┌─────────────────────────────────────────────────────────┐
│                    指示方法4つ                            │
├──────────────┬──────────────────────────────────────────┤
│  常時適用    │  CLAUDE.md（ルール）        ← 毎セッションロード│
│              │  Memory（記憶）            ← 自動参照      │
├──────────────┼──────────────────────────────────────────┤
│  呼び出し時  │  スキル (.claude/skills/)  ← 自然言語マッチング│
│  適用        │  コマンド (~/.claude/commands/) ← /コマンド│
└──────────────┴──────────────────────────────────────────┘
```

間違えると:
- ルールに入れるべきものをスキルにすると → 毎回呼び出す必要があり忘れてしまう
- スキルにすべきものをルールに入れると → CLAUDE.mdが肥大化してトークンの無駄遣い

---

## 判断基準のコア

### 一文の原則

> **「常に守るべき行動原則か？」** → CLAUDE.md ルール
> **「毎回同じ手順を踏む必要があるか？」** → スキル / コマンド

### 判断フロー

```
この指示が必要な状況が来た
       │
       ▼
毎回自動的に適用される必要があるか？
       │
  ┌────┴────┐
  YES       NO
  │         │
  ▼         ▼
ルール       手順が定型化されているか？
(CLAUDE.md)    │
          ┌───┴───┐
          YES     NO
          │       │
          ▼       ▼
        スキル/   普通の会話で
        コマンド  解決できる
```

### 具体的な判断表

| 基準 | → ルール（CLAUDE.md） | → スキル/コマンド |
|------|-------------------|--------------|
| **頻度** | 毎セッション、毎回の返答 | 特定のタイミングのみ（デプロイ前、セッション終了時） |
| **判断の必要性** | なし（必ず従う） | あり（ユーザーが呼び出しを決定） |
| **手順の複雑さ** | 1行〜数行 | 複数ステップ、エージェントの組み合わせ |
| **コンテキスト依存** | なし（常に同じ） | あり（状況ごとに入力が異なる） |
| **成果物** | なし（行動方式の変更） | あり（レポート、ドキュメント、チェックリスト） |

## 実践事例: このセッションでの判断

### Case 1: 「フロントエンドのコードを先に読め」→ ルール

最初は `/frontend-prompt` スキルとして作るか迷いましたが、最終的に**CLAUDE.mdのルール**として決定。

```
IMPORTANT: フロントエンド関連の議論の際は、必ずvista-sphere-proの
git log、コンポーネント構造を先に読んで把握してから回答する。推測しない。
```

**なぜルールか？**
- 「フロントエンドを修正して」と言うたびに `/frontend-prompt` を呼び出すのは面倒
- フロントエンド関連の会話が出たら**必ず**コードを先に読む必要がある
- 呼び出すかどうかをユーザーが決める必要がない（無条件に適用）
- 定型化された手順ではなく行動原則

### Case 2: 「セッション終了時にまとめる」→ スキル（/wrap）

```
4つのエージェントを並列実行 → 重複排除 → ユーザー選択 → 実行
```

**なぜスキルか？**
- 毎セッションではなく**セッション終了時にのみ**必要
- 4つのエージェントを並列実行するという**定型化された手順**がある
- 結果を見てユーザーが**どの更新をするか選択**する必要がある
- 明確な成果物がある（作業サマリー、次にやること、更新提案）

### Case 3: 「コード、コミットは英語で」→ ルール

```
IMPORTANT: コード、コミットメッセージ、CLIの出力などの技術的な成果物は必ず英語を維持
```

**なぜルールか？**
- すべてのコード作成、すべてのコミットに**例外なく**適用
- 判断不要（必ず英語）
- 手順ではなく原則

### Case 4: 「デプロイ前に6項目チェック」→ スキル（/preflight）

```
Git → 環境変数 → DBマイグレーション → CORS → ヘルスチェック → API変更 → PASS/FAIL
```

**なぜスキルか？**
- デプロイのときのみ必要（毎日ではない）
- 6項目を順番に確認する**定型化された手順**
- 結果に応じてpushするかどうかを**ユーザーが決定**
- PASS/FAILチェックリストという成果物

## コマンド vs スキル — どこに置くか

スキル/コマンドとして作ることに決めたら、次の判断:

```
このスキルはすべてのプロジェクトで役に立つか？
       │
  ┌────┴────┐
  YES       NO
  │         │
  ▼         ▼
コマンド    スキル
(~/.claude/  (プロジェクト/.claude/
 commands/)   skills/)
```

| 配置 | 場所 | 呼び出し | 例 |
|------|------|------|------|
| **グローバルコマンド** | `~/.claude/commands/wrap.md` | `/wrap`（どこからでも） | `/wrap`, `/switch`, `/weekly` |
| **プロジェクトスキル** | `プロジェクト/.claude/skills/preflight/SKILL.md` | `/preflight`（そのプロジェクト内のみ） | `/preflight` |

### 追加の違い

| | コマンド | スキル |
|---|---------|-------|
| **frontmatter** | 不要 | `name`、`description` が必須 |
| **自然言語トリガー** | 不可（`/コマンド`のみ） | 可能（description ベースのマッチング） |
| **引数の渡し方** | `$ARGUMENTS` 変数 | 不可 |
| **ファイル構造** | 単一の `.md` ファイル | フォルダ + `SKILL.md`（+ references/） |

## CLAUDE.md のレベル — どの CLAUDE.md に入れるか

ルールに決まったら、どのレベルのCLAUDE.mdに入れるかを決定します:

```
~/.claude/CLAUDE.md              ← すべてのプロジェクトに適用（ペルソナ、言語、トーン）
プロジェクト/CLAUDE.md           ← このプロジェクトにのみ適用（インフラ、規約）
```

| ルールの内容 | レベル | 理由 |
|-----------|------|------|
| 「日本語で答えて」 | グローバル | すべてのプロジェクト共通 |
| 「mainへのpush = 本番デプロイ」 | プロジェクト | WICHIのみに該当 |
| 「.envの内容を出力禁止」 | グローバル | すべてのプロジェクト共通のセキュリティ |
| 「フロントエンドのコードを先に読め」 | プロジェクト | WICHIのフロントエンドのみに該当 |
| 「DB変更前にSQLを見せて」 | グローバル | すべてのプロジェクト共通の安全策 |

## 実践的な判断練習

以下の状況でルール vs スキル、どこに置くかを考えてみましょう:

| 状況 | 判断 | 理由 |
|------|------|------|
| 「PRを上げるたびに変更サマリーを作って」 | コマンド `/pr-summary` | 定型手順 + すべてのプロジェクト共通 |
| 「このプロジェクトはpnpmのみ使う」 | プロジェクト CLAUDE.md | 常時適用 + プロジェクト特化 |
| 「エラーが出たらログを先に確認して」 | グローバル CLAUDE.md | 常時適用 + すべてのプロジェクト共通 |
| 「週次コストレポートを生成」 | プロジェクト スキル | 定型手順 + プロジェクト特化 |
| 「テストなしでコミットしない」 | プロジェクト CLAUDE.md | 常時適用 + プロジェクトごとにテスト方法が異なる |

---

## まとめ

```
指示が必要
    │
    ├─ 常時適用？ ──→ ルール（CLAUDE.md）
    │                  ├─ すべてのプロジェクト？ → グローバル CLAUDE.md
    │                  └─ このプロジェクトのみ？ → プロジェクト CLAUDE.md
    │
    └─ 特定のタイミング？ ──→ スキル/コマンド
                           ├─ すべてのプロジェクト？ → コマンド（~/.claude/commands/）
                           └─ このプロジェクトのみ？ → スキル（.claude/skills/）
```

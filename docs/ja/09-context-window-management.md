# コンテキストウィンドウの実践的な管理 — 200kトークンの現実

## コンテキストウィンドウとは？

Claudeが一度に「見ることができる」テキストの総量が**コンテキストウィンドウ**です。Claude Codeは現在約**200kトークン**（約15万語、日本語基準では約7〜8万語）をサポートしています。

数字だけ見ると非常に多く感じますが、実際に使っていると思ったよりも早く埋まります。

## ノーコードツールとの比較

| ノーコードツール | Claude Code |
|---|---|
| 実行履歴が別途保存される | すべての会話がコンテキストウィンドウに蓄積される |
| 以前の実行結果にアクセス可能 | セッションが終わると会話内容が消える |
| データ容量制限はあるが会話長の制限はない | **会話が長くなると性能が低下** |

これがClaude Codeのすべての記憶システム（CLAUDE.md、Memory、スキル、サブエージェント）が存在する理由です。**有限なコンテキストを効率的に使うための設計**です。

## 200kトークンは実際にどのくらい？

### トークン消費の感覚

| 項目 | おおよそのトークン数 |
|---|---|
| 日本語1文字 | 約1〜3トークン |
| 英語1単語 | 約1トークン |
| コード1行 | 約10〜20トークン |
| 一般的な会話メッセージ1件 | 約50〜200トークン |
| 500行のコードファイル1つ | 約5,000〜10,000トークン |
| Claudeの長い返答1件 | 約500〜3,000トークン |
| CLAUDE.md（簡潔に記述） | 約200〜500トークン |

### コンテキストが早く埋まるケース

```
危険なシナリオ:
  "このプロジェクト全体のコードを全部読んで"
  → ファイル20個 × 平均5,000トークン = 100,000トークン（半分消費）
  → 以降の会話に使えるスペースが半分しか残らない

安全なシナリオ:
  "src/auth/ フォルダのログイン関連ファイルだけ読んで"
  → ファイル3個 × 5,000トークン = 15,000トークン（7.5%消費）
  → 余裕を持って会話を続けられる
```

## コンテキスト消費の構造

セッションが開始されると、コンテキストは以下のように埋まっていきます：

```
200kトークン全体
┌──────────────────────────────────────────┐
│ ██░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ │
│ ↑                                        │
│ システムプロンプト                          │
│ (CLAUDE.md + Auto Memory)                │
│ 約1,000〜3,000トークン                    │
│                                          │
│ 残りスペース: ~197,000トークン             │
│ → 会話 + ファイル読み込み + スキルロードに使用│
└──────────────────────────────────────────┘

会話が進んだ後:
┌──────────────────────────────────────────┐
│ ████████████████████████████░░░░░░░░░░░░ │
│ ↑ システム   ↑ 会話履歴    ↑ 空きスペース  │
│                                          │
│ 会話が長くなるほど空きスペースが減る         │
└──────────────────────────────────────────┘
```

## コンテキストが不足したときに起こること

Claude Codeはコンテキストの限界に達すると**自動圧縮**を実行します：

```
コンテキスト限界に到達
  │
  ▼
以前の会話を自動的に要約/圧縮
  │
  ▼
要約に置き換え → スペースを確保
  │
  ▼
会話を継続可能（ただし、詳細情報が失われる可能性あり）
```

**問題点**: 圧縮処理の過程で以前の会話の詳細が失われることがあります。「さっき3番目のファイルの27行目の変数名は何だっけ？」といった質問に正確に答えられなくなることがあります。

## 実践的なトークン節約戦略

### 1. 範囲を絞って質問する

```
❌ "このプロジェクト全体の構造を分析して"
   → すべてのファイルを読もうとする → トークンが爆発

✅ "src/auth/ フォルダのログインフローを分析して"
   → 関連ファイルのみ読み込む → トークンを節約
```

### 2. 長い作業はサブエージェントに委任する

```
❌ メインの会話で "100個のファイルを全部確認して"
   → メインのコンテキストがいっぱいになる

✅ "100個のファイルを確認して問題のあるものだけ教えて"
   → サブエージェントが独立したスペースで処理 → 結果の要約のみメインに返す
```

### 3. 新しいセッションに切り替える

```
状況: 会話が50ターンを超え、Claudeの返答が遅くなったり不正確になっている

対処法:
  1. 現在までの結論/決定事項をまとめるよう依頼
  2. 重要な内容はCLAUDE.mdやファイルに記録
  3. 新しいセッションを開始（Ctrl+C → claude を再実行）
  4. クリーンなコンテキストで作業を継続
```

### 4. CLAUDE.mdを簡潔に保つ

```
❌ CLAUDE.mdにプロジェクト全体のドキュメントをコピー
   → 毎セッション数千トークンを消費

✅ CLAUDE.mdにはコアとなるルールだけを記述し、詳細はスキルに分離
   → 必要なときだけロードしてトークンを節約
```

### 5. `/clear` コマンドを活用する

```
会話の途中で話題が完全に変わるとき:
  /clear → 会話履歴を初期化（CLAUDE.md、Memoryは維持）
  → クリーンな状態で新しいトピックを開始
```

## 機能ごとのトークン影響

| 機能 | トークンへの影響 | 管理戦略 |
|---|---|---|
| **CLAUDE.md** | 毎セッション常時占有 | 短く保つ（500トークン以内を推奨） |
| **Auto Memory** | 毎セッション常時占有 | 200行以内に自動制限される |
| **スキル** | 呼び出し時のみ占有 | 長いスキルでもOK（オンデマンド） |
| **ファイル読み込み** | 読んだ分だけ占有 | 必要なファイルのみ指定して読む |
| **サブエージェント** | メインには結果のみ返す | 大量処理に積極的に活用 |
| **会話履歴** | 累積して占有 | 定期的に/clearまたは新しいセッション |

## コンテキスト不足のサインと対処法

| サイン | 意味 | 対処法 |
|---|---|---|
| 返答速度が遅くなる | コンテキストが埋まって処理量が増加 | 新しいセッションを開始 |
| 以前の会話内容を間違えて記憶する | 自動圧縮で詳細が失われた | 重要内容をファイルに記録して新しいセッション |
| "コンテキスト制限"のようなメッセージ | 限界に到達 | 即座に新しいセッションに切り替える |
| 同じファイルを繰り返し読もうとする | 以前読んだ内容が圧縮で失われた | サブエージェントへの委任を検討 |

## まとめ

- コンテキストウィンドウ200kトークン = Claudeが一度に見ることができる総量
- すべて（システムプロンプト + 会話 + ファイル読み込み）がこのスペースを共有
- 会話が長くなると自動圧縮 → 詳細が失われる可能性あり
- **節約戦略**: 範囲を絞る、サブエージェントへの委任、新しいセッションへの切り替え、CLAUDE.mdを簡潔に保つ
- Claude Codeの記憶システム（CLAUDE.md、Memory、スキル、サブエージェント）はすべてこの制約を克服するための設計

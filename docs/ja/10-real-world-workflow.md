# 実践ワークフロー — Claude Code活用パターン集

## このドキュメントの目的

前のドキュメントでClaude Codeの個別の概念（システム、記憶、スキル、サブエージェント、MCP、フックなど）を学びました。このドキュメントでは、**実際の業務でこれらの概念をどのように組み合わせて使うのか**をパターン別に整理します。

## Pattern 1: CLAUDE.md 作成戦略

### グローバル CLAUDE.md — 個人の作業スタイル設定

```markdown
# ~/.claude/CLAUDE.md

## Language
- デフォルトの返答言語: 日本語
- コード、コミットメッセージ、CLIの出力: 英語を維持

## Tone
- 簡潔で論理的なトーン
- 不要な挨拶は省略

## Preferences
- 構造化した回答の場合はテーブルを優先使用
- コード変更時は変更理由を簡単に説明
```

**ポイント**: このファイルはすべてのプロジェクトに適用されるため、**言語/トーン/形式**のような個人的な好みのみ記述します。プロジェクト別のルールは入れません。

### プロジェクト CLAUDE.md — チームの規約

```markdown
# ./CLAUDE.md

## Project
- Next.js 14 + TypeScript + Tailwind CSS
- パッケージマネージャー: pnpm

## Conventions
- コンポーネント: PascalCase (UserProfile.tsx)
- APIルート: kebab-case (user-profile/)
- コミットメッセージ: Conventional Commits (feat:, fix:, docs:)

## Rules
- 新機能追加時は必ずテストコードを含める
- console.logのコミット禁止
- any型の使用禁止
```

**ポイント**: チームメンバー全員が共有するルールです。Gitに含めてチームの規約として活用します。

### CLAUDE.md 作成ミスの例

| ミス | なぜ問題か | 代替案 |
|---|---|---|
| プロジェクト全体のドキュメントをコピーして貼り付ける | 毎セッション数千トークンを消費 | コアルールのみ、詳細はスキルに分離 |
| "可能であれば〜してください"のような曖昧な表現 | Claudeが無視することがある | "必ず〜すること" という命令形を使用 |
| 矛盾するルールが複数箇所に存在 | 予測不能な動作 | 一箇所でのみ定義し、優先順位を明示 |
| ルールが多すぎる（50件以上） | すべてに従うことが難しい | コアとなる10件以内に制限 |

## Pattern 2: 日常業務の自動化

### 朝のルーティン スキル

```markdown
# .claude/skills/morning-check/SKILL.md

## 役割
プロジェクトの状態を確認するアシスタント

## やること
1. git logで昨日以降のコミットを確認
2. git statusで未完了の作業を確認
3. TODO/FIXMEコメントを検索
4. 以下のフォーマットで整理:

### 昨日のコミット
- [一覧]

### 進行中（未コミットの変更）
- [一覧]

### 残りのTODO
- [ファイル:行] 内容
```

**使用方法**: 毎朝 `/morning-check` を実行 → プロジェクトの現状を把握

### ノーコードツールとの比較

| 従来の方法 | Claude Codeの方法 |
|---|---|
| Notionに手動で日次記録を作成 | `/morning-check`で自動生成 |
| Jiraダッシュボードでイシューを確認 | Gitベースで実際のコード変更を追跡 |
| Slackのスタンドアップボットに直接入力 | 自動分析後に整理版を提供 |

## Pattern 3: 学習とドキュメント化のワークフロー

### 新しいコードベースの分析

```
Step 1: 全体構造の把握（Explore サブエージェントを活用）
  "このプロジェクトのディレクトリ構造と主要な技術スタックを分析して"

Step 2: コアフローの追跡
  "ユーザーがログインするときにコードがどの順序で実行されるかを追跡して"

Step 3: ドキュメント化
  "分析結果をdocs/architecture.mdとしてまとめて"
```

**ポイント**: Step 1でExplore サブエージェントが自動的に活用されます。全ファイルをメインのコンテキストに乗せずに独立したスペースで探索 → 要約のみを返します。

### 学習ノート作成パターン

```
Step 1: 概念の質問
  "Next.jsのApp RouterとPages Routerの違いを説明して"

Step 2: 実習コードの生成
  "簡単なサンプルコードで違いを見せて"

Step 3: 整理と保存
  "今までの内容をdocs/nextjs-routing.mdとしてまとめて"
```

## Pattern 4: コードレビューの自動化

### スキル + サブエージェントの組み合わせ

```markdown
# .claude/skills/review/SKILL.md

## 役割
シニア開発者視点のコードレビュアー

## やること
1. 最近のコミットの変更点を分析
2. 以下の観点でレビュー:
   - セキュリティ: SQLインジェクション、XSSなど
   - パフォーマンス: 不要な演算、N+1クエリ
   - 可読性: 命名規則、関数の長さ
   - テスト: 不足しているテストケース
3. 深刻度別に分類してテーブルで出力

## 出力形式
| ファイル | 行 | 深刻度 | カテゴリ | 内容 |
```

**上級活用**: Agent Teamsが有効な場合、セキュリティ/パフォーマンス/可読性の専門家をそれぞれ別のエージェントとして起動して並列レビューが可能です。

## Pattern 5: 定期レポートの生成

### MCP + スキルの組み合わせ

```markdown
# .claude/skills/weekly-report/SKILL.md

## やること
1. GitHub MCPで今週マージされたPRの一覧を取得
2. 各PRの変更内容の要約を作成
3. 今週作成/解決されたイシューの統計を整理
4. docs/reports/ フォルダに週次レポートを保存
5. （MCP設定済みの場合）Slack #team-updates チャンネルに要約を送信

## レポート形式
# 週次開発レポート (YYYY-MM-DD)

## 完了項目
| PR | 作成者 | 要約 |

## イシュー状況
- 新規: N件
- 解決: N件
- 未解決: N件

## 来週の予定
- [予定項目]
```

## Pattern 6: 安全な自動化の構築

### フック + カスタムサブエージェントの組み合わせ

```
自動化の階層:

1. フック（自動実行）
   └─ ファイル保存時 → Prettier + ESLint を自動実行
   └─ 危険なコマンド → 自動ブロック

2. カスタムサブエージェント（制限された権限）
   └─ DBクエリ専用（Read、Grepのみ許可、Write/Bash禁止）
   └─ ログ分析専用（特定のディレクトリのみアクセス可）

3. スキル（手動呼び出し）
   └─ デプロイチェックリスト（人が確認してから実行）
```

**ポイント**: 自動化のレベルに応じてツールを階層的に使用します。完全自動（フック）→ 制限付き自動（サブエージェント）→ 手動（スキル）。

## Pattern 7: PCなしでClaude Codeを活用する

Claude Codeはターミナルで実行されるため、PCがなければ使えないと思いがちです。しかし、いくつかの方法でモバイルからもアクセスできます。

### Chat Interface vs CLI — まず知っておくべきこと

claude.aiアプリ（Chat Interface）とClaude Code（CLI）は**まったく別の製品**です。会話履歴、設定、ファイルはすべて共有されません（→ [01-claude-code-体계.md](01-claude-code-体계.md) 参照）。したがって、モバイルで「Claude Codeと同じ体験」を望む場合、claude.aiアプリを開くのではなく、**CLI自体にリモートアクセス**する必要があります。

### 方法別の比較

| 方法 | 必要条件 | CLI機能の使用 | 推奨状況 |
|------|----------|:------------:|----------|
| **Claudeモバイルアプリ** | なし | X（会話のみ） | 簡単な質問、ブレインストーミング |
| **SSH + モバイルターミナル** | PC電源ON + SSH設定 | O（完全） | コード作業の継続 |
| **GitHub Codespaces** | GitHubアカウント | O（完全） | PCを長期間使わない場合 |

### SSHリモートアクセスのセットアップ（推奨）

最も実用的な組み合わせ: **Tailscale（無料VPNメッシュ）+ モバイルターミナルアプリ**

```
[モバイル]                         [PC（自宅/オフィス）]
Termius / Blink Shell  ───→  SSHサーバー（リモートログイン有効）
        │                              │
        └──── Tailscale VPN ───────────┘
               （ポートフォワード不要、どこからでも安全に接続）
```

設定手順:
1. PC: システム設定 → 一般 → 共有 → **リモートログイン**を有効化
2. PC + モバイルの両方に**Tailscale**をインストールし、同じアカウントでログイン
3. モバイルターミナルアプリ（Termiusなど）でTailscaleが割り当てたIPでSSH接続
4. ターミナルで `claude` を実行 → PCにいるときと同じ環境で作業

**Tailscaleを使う理由**: 一般的なSSHはルーターのポートフォワード、固定IPなどのネットワーク設定が複雑です。Tailscaleはインストールするだけで機器間の仮想プライベートネットワークを自動構成してくれるため、カフェや電車の中などどこからでも自宅のPCに直接接続できます。

### モバイルターミナルアプリの選択肢

| アプリ | プラットフォーム | 特徴 |
|----|--------|------|
| **Termius** | iOS / Android | シンプルなUI、無料プランで十分 |
| **Blink Shell** | iOS | プロ向け、moshサポート |
| **JuiceSSH** | Android | 軽量で高速 |

### ChatとCLIの両方で同じコンテキストを維持する

2つの製品間で履歴が共有されないため、コンテキストを維持するには手動戦略が必要です：

| 戦略 | 方法 |
|------|------|
| **共有ナレッジベース** | Notionなどにコアコンテキストを整理。CLIはMCPでNotion連携、Chatは手動参照 |
| **セッション引き継ぎ** | CLI作業後に要約をファイルに保存 → Chatでその要約を貼り付けて会話を継続 |
| **SSH直接アクセス** | モバイルからCLI自体を使用 → 履歴の問題自体が発生しない（最もクリーン） |

## ワークフロー設計ガイド

### どのツールをいつ使うか

```
"この作業を自動化したい"
  │
  ├─ 毎回同じ単純な処理？ ──▶ フック
  │   （フォーマット、リント、ブロック）
  │
  ├─ 繰り返しだが判断が必要？ ──▶ スキル
  │   （レポート生成、コードレビュー）
  │
  ├─ 外部サービスの連携が必要？ ──▶ MCP + スキル
  │   （Slack送信、DBクエリ）
  │
  ├─ 大量データの処理？ ──▶ サブエージェント
  │   （100ファイルの分析、大規模なログ解析）
  │
  └─ 複数の専門家の視点が必要？ ──▶ Agent Teams
      （並列レビュー、同時開発）
```

### ノーコードツール → Claude Code 移行マッピング

| 既存のノーコードワークフロー | Claude Codeへの移行 |
|---|---|
| Zapier: Gmail受信 → Notionに整理 | MCP（Gmail）+ スキル（整理ロジック） |
| Make: 毎週月曜 → レポート生成 → Slack送信 | スキル（レポート）+ MCP（Slack） |
| GitHub Actions: PR → 自動レビュー | フック（PRイベント）+ スキル（レビューロジック） |
| Notion自動化: ステータス変更 → 通知 | フック + MCP（Notion） |

## まとめ

- **CLAUDE.md**: グローバル（個人の好み）/ プロジェクト（チームルール）/ ローカル（個人設定）で階層的に分離
- **日常自動化**: スキルで繰り返し作業をレシピ化
- **安全な自動化**: フック（自動）→ サブエージェント（制限付き自動）→ スキル（手動）の階層設計
- **外部連携**: MCP + スキルの組み合わせでノーコードツールレベルのワークフローを実現
- **モバイル活用**: PCなしでもSSH（Tailscale + ターミナルアプリ）でCLIにリモートアクセス可能。ChatとCLIは別物なのでコンテキスト維持の戦略が必要
- ツール選択基準: 単純な繰り返し（フック）/ 判断が必要（スキル）/ 外部連携（MCP）/ 大量処理（サブエージェント）/ 多視点（Teams）
